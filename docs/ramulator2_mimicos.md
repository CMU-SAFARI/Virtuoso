# Ramulator2 + MimicOS Integration Guide

This document describes how Virtuoso integrates the [Ramulator2](https://github.com/CMU-SAFARI/ramulator2) cycle-accurate DRAM simulator with MimicOS, the shared userspace memory management kernel. This integration enables accurate modeling of DRAM timing during page table walks, TLB simulations, and physical memory allocation.

## Overview

In Virtuoso's Sniper-based flow, DRAM accesses during page table walks are modeled using Sniper's internal timing model. For studies requiring higher-fidelity DRAM behavior (e.g., RowHammer, DRAM refresh interference, bank-level parallelism during walks), Virtuoso provides a Ramulator2 integration path.

MimicOS communicates with Ramulator2 via an **IPC bridge** consisting of:
- **Named pipes** for request/response communication
- An **Intel SDE pintool** that intercepts memory operations from the traced application
- **MimicOS** running as a subprocess that handles page faults and physical memory allocation

Ramulator2 models the DRAM timing of every memory access, including those generated by page table walks, while MimicOS ensures that the physical address mappings and allocation policies are consistent with the Sniper-based simulations.

### Data Flow

1. **Ramulator2** drives the simulation by replaying a memory trace.
2. On each memory access, the **TLB simulation** within Ramulator2 checks for a hit.
3. On a TLB miss, a **page walk** is initiated. Each level of the walk generates a DRAM access that is timed by Ramulator2's DRAM model.
4. If the virtual page has no mapping (first touch), a **page fault** is signaled via IPC to the SDE pintool.
5. The pintool forwards the fault to **MimicOS**, which allocates a physical frame, updates the page table, and returns the mapping.
6. The walk completes with the new mapping, and Ramulator2 records the total latency including DRAM timing for all walk levels.

## Prerequisites

- **CMake** 3.14+ and a C++17-capable compiler (g++ 9+ or clang 10+)
- **Intel SDE** (Software Development Emulator) -- the pintool IPC bridge requires the SDE build kit
- **Python 3.8+** for configuration and scripting
- Virtuoso repository cloned with submodules initialized

Ensure the Ramulator2 submodule is present:
```bash
git submodule update --init simulator/ramulator2
```

## Setup

### Step 1: Apply the MimicOS Integration Patch

From the repository root:

```bash
bash patches/apply_ramulator2_mimicos.sh
```

This script:
1. Applies `patches/ramulator2_mimicos.patch` to the Ramulator2 submodule, adding translation modeling, TLB simulation, and pintool sources.
2. Creates a symlink from `simulator/ramulator2/mimicos` to the shared `mimicos/` directory at the repository root.
3. Verifies that shared headers (e.g., `mimicos/include/mm/allocator_factory.h`) are accessible.

### Step 2: Build Ramulator2

```bash
cd simulator/ramulator2
mkdir -p build && cd build
cmake ..
make -j
```

The patched CMakeLists.txt automatically picks up the MimicOS sources and the translation/TLB modules.

### Step 3: Build the SDE Pintool

```bash
cd simulator/ramulator2/pintool
make SDE_BUILD_KIT=/path/to/your/sde-external-kit
```

Replace `/path/to/your/sde-external-kit` with the actual path to your Intel SDE installation directory.

### Step 4: Build MimicOS Standalone

```bash
cd simulator/ramulator2/mimicos
make SNIPER_INCLUDE=$PWD/../../sniper/include
```

The `SNIPER_INCLUDE` variable points MimicOS to the simulator-agnostic allocator template headers (e.g., `physical_memory_allocator.h`, `reserve_thp.h`, `spot.h`) shared between Sniper and Ramulator2.

## Configuration

### Ramulator2 Configuration

Ramulator2's YAML configuration files control the DRAM organization, timing, and controller policy. Example:

```yaml
# example_config.yaml
Frontend:
  impl: TraceBasedFrontend
  path: ./example_inst.trace

MemorySystem:
  impl: GenericDRAM
  clock_ratio: 1

  DRAM:
    impl: DDR4
    org:
      preset: DDR4_8Gb_x8
    timing:
      preset: DDR4_2400

  Controller:
    impl: Generic
    plugins:
      - impl: TraceRecorder
```

### TLB Configuration

The patched Ramulator2 accepts an optional TLB configuration file that mirrors Sniper's TLB hierarchy:

```yaml
# test_tlb_config.yaml
TLB:
  levels: 2
  L1:
    entries: 64
    associativity: 4
    page_size: 4096
    latency: 1
  L2:
    entries: 1536
    associativity: 12
    page_size: 4096
    latency: 8
  PWC:
    enabled: true
    levels: 3
    entries: [32, 32, 32]
    latency: [2, 2, 2]
```

### MimicOS Configuration

MimicOS accepts an INI-style configuration that selects the allocator and page table:

```ini
[memory]
allocator = reserve_thp
page_table = radix_4level
physical_memory_mb = 16384

[reserve_thp]
target_fragmentation = 0.0
huge_page_size = 2097152
```

## Running Simulations

### Basic Run

```bash
cd simulator/ramulator2

# Run with Ramulator2 driving the trace, MimicOS handling faults
./build/ramulator2 \
  --config example_config.yaml \
  --tlb-config test_tlb_config.yaml \
  --mimicos-config mimicos/configs/default.ini
```

### With SDE Pintool (Full IPC Bridge)

For the full flow with the SDE pintool intercepting a live application:

```bash
# Terminal 1: Start MimicOS as a daemon
cd simulator/ramulator2/mimicos
./mimicos --config configs/default.ini --ipc-mode pipe --pipe-path /tmp/mimicos_pipe

# Terminal 2: Start Ramulator2 with pintool-driven frontend
cd simulator/ramulator2
/path/to/sde -t ./pintool/obj-intel64/mimicos_bridge.so \
  --pipe-path /tmp/mimicos_pipe \
  -- ./build/ramulator2 --config example_config.yaml
```

### Using the Wrapper Script

A convenience wrapper handles process orchestration:

```bash
cd simulator/ramulator2
python3 mimicos/run_mimicos_wrapper.py \
  --ramulator-config example_config.yaml \
  --mimicos-config mimicos/configs/default.ini \
  --trace example_inst.trace
```

## IPC Protocol

MimicOS and Ramulator2 communicate via a simple request-response protocol over named pipes. The message types are:

### Message Flow

```
Ramulator2                              MimicOS
    │                                      │
    │──── PAGE_FAULT_REQUEST(vaddr) ──────►│
    │                                      │  (allocate physical frame)
    │                                      │  (update page table)
    │◄─── MEMORY_ACCESS(paddr, level) ─────│
    │                                      │
    │  (model DRAM timing for walk)        │
    │                                      │
    │──── FAULT_RESPONSE(latency) ────────►│
    │                                      │
    │  ... (simulation continues) ...      │
    │                                      │
    │──── MIMICOS_DONE ───────────────────►│
    │                                      │  (flush stats, clean up)
```

### Message Types

| Message | Direction | Payload | Description |
|---------|-----------|---------|-------------|
| `PAGE_FAULT_REQUEST` | Ramulator2 -> MimicOS | Virtual address, access type | A TLB miss triggered a page walk that found no mapping |
| `MEMORY_ACCESS` | MimicOS -> Ramulator2 | Physical address, walk level | MimicOS tells Ramulator2 to model the DRAM access for this page table entry |
| `FAULT_RESPONSE` | Ramulator2 -> MimicOS | DRAM latency (cycles) | Ramulator2 reports the measured DRAM latency back to MimicOS |
| `MIMICOS_DONE` | Ramulator2 -> MimicOS | -- | Simulation complete; MimicOS flushes statistics and shuts down |

### Protocol Details

- All messages are fixed-size binary structs for minimal parsing overhead.
- The pipe is full-duplex: Ramulator2 and MimicOS alternate as sender/receiver.
- On `PAGE_FAULT_REQUEST`, MimicOS may issue **multiple** `MEMORY_ACCESS` messages (one per page table level) before returning to normal flow.
- MimicOS tracks per-page statistics (access count, allocation timestamp) and dumps them at `MIMICOS_DONE`.

## Shared MimicOS Architecture

A key design principle of Virtuoso is that **MimicOS is a single, shared codebase** used by both Sniper and Ramulator2. This ensures that:

1. **Allocation decisions are identical**: The same `ReserveTHP`, `SpOT`, `Utopia`, or other allocator code runs regardless of which simulator drives the workload.
2. **Page table state is consistent**: Page table structures and walk logic are shared, so results are directly comparable.
3. **New features propagate automatically**: Adding a new allocator or page table to `mimicos/` makes it available to both simulation backends without duplication.

### How Sniper Uses MimicOS

In Sniper, MimicOS runs **in-process** as part of the simulation. The `ApplicationContext` class in Sniper's memory management subsystem instantiates MimicOS components directly:
- Physical allocator (selected by config)
- Page table (radix by default)
- Exception handlers (page fault, protection fault)

Sniper's MMU designs call into MimicOS on TLB misses to perform page walks and, when needed, trigger page fault handling.

### How Ramulator2 Uses MimicOS

In Ramulator2, MimicOS runs as a **separate process** communicating over IPC. The integration patch adds:
- A `TranslationFrontend` module to Ramulator2 that intercepts memory accesses and performs TLB lookups.
- An IPC client that sends `PAGE_FAULT_REQUEST` messages to the MimicOS process.
- A DRAM-timing-aware page walk model that issues `MEMORY_ACCESS` requests for each page table level.

The same MimicOS binary serves both modes -- it detects whether it is running in-process (Sniper) or over IPC (Ramulator2) based on its startup configuration.

### Adding a New Allocator

To add a new physical memory allocator that works with both Sniper and Ramulator2:

1. **Create the allocator template** in `simulator/sniper/include/memory_management/physical_memory_allocators/my_allocator.h`. This header should be simulator-agnostic (no Sniper or Ramulator2 dependencies).

2. **Implement the allocator** in `mimicos/src/physical_allocator/my_allocator.cpp`, using only the MimicOS API.

3. **Register in the factory** by adding an entry to `mimicos/include/mm/allocator_factory.h`.

4. **Create a config module** at `simulator/sniper/config/physical_memory_allocators/my_allocator.cfg`.

5. **Compose into a scheme** by creating a new `address_translation_schemes/my_scheme.cfg` that `#include`s your allocator config.

Both Sniper and Ramulator2 will automatically pick up the new allocator.

## Troubleshooting

### Common Issues

**"Shared MimicOS headers not found"**
The apply script could not find `mimicos/include/mm/allocator_factory.h`. Ensure the `mimicos/` directory exists at the repository root and contains the expected headers.

**"SNIPER_INCLUDE not set"**
When building MimicOS standalone for Ramulator2, you must provide the path to Sniper's `include/` directory so that the shared allocator templates are found:
```bash
make SNIPER_INCLUDE=/path/to/Virtuoso/simulator/sniper/include
```

**Ramulator2 build fails with missing translation headers**
Make sure you ran `bash patches/apply_ramulator2_mimicos.sh` before building. The patch adds the necessary source files to Ramulator2.

**IPC pipe errors**
Ensure both MimicOS and Ramulator2 agree on the pipe path. Delete stale pipe files before restarting:
```bash
rm -f /tmp/mimicos_pipe*
```

**Submodule not initialized**
```bash
git submodule update --init simulator/ramulator2
```
